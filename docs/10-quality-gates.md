# Divine Punishment — 品質ゲート定義書

作成日: 2026-07-04
ステータス: Draft v1

指示書（文書）だけでは担保できない品質 — レベルの面白さ・アートの質感・操作の手触り — を**検証ゲート**で担保する。各ゲートは「判定者・タイミング・合否基準・記録」を持ち、**不合格の場合は次工程に進めない**。

## 役割定義

| 役割 | 担当 | 責務 |
|---|---|---|
| 実装エージェント | Issue ごとにアサイン | 指示書に従い実装・自己検証・ゲート申請 |
| レビューエージェント | 実装と**別の**エージェント | PR レビュー（G1）・ゲート判定の実施と記録 |
| プロダクトオーナー（PO） | ユーザー（人間） | G4 の承認・G3/G5 の抜き取り確認・仕様変更の最終承認 |

## ゲート一覧

### G1: コード品質ゲート（全 PR）

| 項目 | 内容 |
|---|---|
| タイミング | 全 PR のマージ前 |
| 判定者 | CI + レビューエージェント |
| 合否基準 | CI green（GUT 全通過 + 3 OS エクスポート）/ CONTRIBUTING §4 の DoD 全チェック / docs/08 §10 との差分なし |
| 記録 | PR 上のレビューコメントと DoD チェックリスト |

### G2: メカニクス品質ゲート（機能 Issue の完了時）

| 項目 | 内容 |
|---|---|
| 対象 | プレイヤーアクション・知覚・AI・忍具など、体感を持つ機能 |
| タイミング | 該当 Issue の PR マージ前（G1 と同時） |
| 判定者 | レビューエージェント（実機で操作して判定） |
| 合否基準 | ①ジムレベル（#13）で該当機能の検証手順（指示書に記載）を再現できる ②デバッグオーバーレイの数値が tuning データと一致 ③10 分の自由操作で引っかかり / すり抜け / 進行不能がない |
| 記録 | `docs/qa-logs/G2-issue-<番号>.md`（手順・結果・気づき） |

### G3: レベル品質ゲート（ミッションごと・2 段階）

| 項目 | 内容 |
|---|---|
| G3a（グレーボックス完了時） | 判定者: レビューエージェント。基準: 07 付録チェックリスト全通過・**経路 3 系統それぞれの未発見クリアを実プレイで達成**（プレイログ添付）・基準時間の実測値が想定 ±30% 以内・「理不尽な発見 / 検知漏れ」ゼロ |
| G3b（アートパス完了時） | 判定者: レビューエージェント + PO 抜き取り。基準: G3a の再実行で退行なし・見た目の明暗と V 値の乖離なし（デバッグオーバーレイ）・60fps 維持 |
| 記録 | `docs/qa-logs/G3-m<ミッション番号>.md` |
| 不合格時 | チューニング or レイアウト修正 Issue を切って再判定。**面白くない（経路選択に意味がない・観察の報酬がない）場合も不合格にしてよい** — その際は 04 §1 のレベルデザイン原則のどれに反するかを明記する |

### G4: ナラティブ / アート品質ゲート（PO 承認必須）

| 項目 | 内容 |
|---|---|
| 対象と基準 | ①カットシーン静止画の**画風サンプル 3 枚**（#77）: PO 承認で画風固定、以降の量産は照合のみ ②隠れ家会話等の**補筆テキスト**: トーンガイド（06 §1.2）のチェックリスト（一人称 / 文長 / 感情語禁止 / 現代語彙禁止）をレビューエージェントが通した後、PO が全文読了して承認 ③**確定台詞**（06 §3 の全脚本）は一字一句変更禁止 — 変更したい場合は PO 承認で 06 を先に改訂 |
| 記録 | `docs/qa-logs/G4-<対象>.md` + PO 承認の発言記録 |

### G5: 総合プレイテストゲート（Phase 節目）

| 項目 | 内容 |
|---|---|
| タイミング | Phase 1 完了時（#41 / #49）と Phase 2 完了時（#83 / #84） |
| 判定者 | 外部テスター（Phase 2 は 2 名以上）+ PO |
| 合否基準 | 01-requirements §4.1 / §4.2 の該当項目。テスターには**報告テンプレ**（下記）で記録してもらう |
| 記録 | `docs/qa-logs/G5-phase<1|2>.md` |

**テスター報告テンプレ**（qa-logs にコピーして使用）

```
ミッション / 区画:
発生事象: （例: 茂みにいたのに発見された）
その時の状況: 天候 / 光源 / スタンス / 敵の警戒段階
期待した挙動:
再現手順（分かれば）:
体感の理不尽度: 1(納得できる) - 5(納得できない)
```

## 運用メモ

- qa-logs は `docs/qa-logs/` にコミットする（Vault ではなくリポジトリ。実装エージェントが参照するため）
- ゲート判定で仕様の穴が見つかった場合: 修正は docs 改訂 → Issue 更新 → 実装 の順。実装先行禁止
- 「合格の水増し」を防ぐため、G2/G3 の判定者は当該 Issue の実装者と別であること
